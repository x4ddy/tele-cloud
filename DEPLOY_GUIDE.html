TeleCloud · Deployment Runbook

Deploy TeleCloud for free — Render + Vercel

Exact, ordered, click-by-click steps to take this repo from local to a live URL. Everything here uses free tiers only.

Backend → Render Frontend → Vercel DB/Auth → Supabase Redis+QStash → Upstash Storage → Telegram
Read the order first. There is a chicken-and-egg: the backend needs the frontend's URL (for CORS + email redirects), and the frontend needs the backend's URL. The trick is a two-pass deploy — deploy the backend first with placeholders, deploy the frontend using the backend URL, then come back and fill the two frontend-dependent values on the backend. The steps below already handle this; just don't skip around.

Step 1 — Create the accounts

1Sign up (all free, no card needed)

  • GitHub — hosts the code; Render & Vercel deploy from it.
  • Supabase — Postgres database + Auth.
  • Upstash — Redis (rate limiting) + QStash (cron cleanup).
  • Render — hosts the FastAPI backend.
  • Vercel — hosts the frontend.
  • Telegram account (for @BotFather) — the file storage backend.
Tip: sign in to Render and Vercel with GitHub — it makes repo import one click.

Step 2 — Provision Supabase (DB + Auth)

2Create the project and grab keys

  1. In the Supabase dashboard → New project. Give it a name and a strong DB password (save it).
  2. Pick a region and remember it. Everything else (Render, Upstash) should sit in the same region — this is the single biggest performance lever (see hiccups).
  3. Wait for provisioning (~2 min), then open Project Settings → API and copy:
    Supabase valueEnv var it becomes
    Project URLSUPABASE_URL
    Project API keys → anon publicSUPABASE_ANON_KEY
    Project API keys → service_role (secret)SUPABASE_SERVICE_ROLE_KEY
    JWT Settings → JWT SecretSUPABASE_JWT_SECRET
  4. Open Project Settings → Database → Connection string and copy the URI (you'll use it once, to run migrations). This is DATABASE_URL — the app itself never needs it.
Never put service_role or the JWT secret in the frontend or in git. The anon key is the only Supabase value that's public by design.

Step 3 — Apply the database migrations

3Run the 5 SQL files — the app 500s without this

The schema, row-level-security policies, and functions live in telecloud/database/migrations/. Nothing runs them for you. Apply them in order.

Easiest way (dashboard): Supabase → SQL Editor → New query. Open each file, paste its contents, click Run, in this exact order:

0001_schema.sql
0002_rls.sql
0003_functions.sql
0004_profiles_verification_token.sql
0005_supabase_email_verification.sql

Or via CLI (from the repo root), using the connection string from Step 2:

psql "$DATABASE_URL" -f telecloud/database/migrations/0001_schema.sql
psql "$DATABASE_URL" -f telecloud/database/migrations/0002_rls.sql
psql "$DATABASE_URL" -f telecloud/database/migrations/0003_functions.sql
psql "$DATABASE_URL" -f telecloud/database/migrations/0004_profiles_verification_token.sql
psql "$DATABASE_URL" -f telecloud/database/migrations/0005_supabase_email_verification.sql
Verify: In Supabase → Table Editor you should now see the app's tables (profiles, folders, files, shares, …). If they're missing, the migrations didn't apply and every authed request will fail with a 500 even though /health is green.

Step 4 — Provision Upstash (Redis + QStash)

4Redis (rate limiting + retry queue)

  1. Upstash console → Redis → Create Database. Choose the same region as Supabase.
  2. On the database page, scroll to REST API and copy:
    Upstash valueEnv var
    REST URL (UPSTASH_REDIS_REST_URL)UPSTASH_REDIS_REST_URL
    REST TokenUPSTASH_REDIS_REST_TOKEN

QStash (scheduled cleanup callbacks)

  1. Upstash console → QStash. On the QStash page find Signing keys and copy both:
    QStash valueEnv var
    Current signing keyQSTASH_CURRENT_SIGNING_KEY
    Next signing keyQSTASH_NEXT_SIGNING_KEY
  2. You'll create the actual scheduled calls in Step 11, after the backend has a URL. The keys above just let the backend verify QStash's signatures.

Step 5 — Set up Telegram bots + channels

5Bots and storage channels

  1. In Telegram, message @BotFather → /newbot. Create at least one bot (more = more upload throughput). Copy each bot token → these become the comma-separated TELEGRAM_BOT_TOKENS.
  2. Create one or more private channels (these physically store the file chunks).
  3. In each channel: Add member → add your bot(s) → promote to Admin (they must be able to post & delete messages).
  4. Get each channel's numeric id (a negative bigint like -1001234567890). Quick way: forward a message from the channel to @userinfobot / @getidsbot, or read it from the Bot API. These become the comma-separated TELEGRAM_CHANNEL_IDS.
Example values:
TELEGRAM_BOT_TOKENS=123456:ABC-token-one,789012:DEF-token-two
TELEGRAM_CHANNEL_IDS=-1001234567890,-1009876543210

Step 7 — Deploy the backend to Render

7Create the web service from the Blueprint

  1. Render dashboard → New → Blueprint → connect your GitHub repo. Render reads render.yaml and proposes a service called telecloud-api.
  2. Region: pick the one closest to your Supabase region (Render free tier offers Oregon, Ohio, Virginia, Frankfurt, Singapore). Matching regions is what keeps requests sub-second.
  3. Render will ask for the sync:false secrets. Fill in everything you gathered:
    • TELEGRAM_BOT_TOKENS, TELEGRAM_CHANNEL_IDS
    • SUPABASE_URL, SUPABASE_ANON_KEY, SUPABASE_SERVICE_ROLE_KEY, SUPABASE_JWT_SECRET
    • UPSTASH_REDIS_REST_URL, UPSTASH_REDIS_REST_TOKEN
    • QSTASH_CURRENT_SIGNING_KEY, QSTASH_NEXT_SIGNING_KEY
  4. For the two frontend-dependent vars, set a temporary placeholder for now — you'll fix them in Step 9 once Vercel gives you a URL:
    • APP_BASE_URL = https://placeholder.vercel.app
    • CORS_ALLOWED_ORIGINS = https://placeholder.vercel.app
    (APP_ENV=production and PYTHON_VERSION=3.12.4 are already baked into render.yaml.)
  5. Click Apply / Create. Render runs pip install -r requirements.txt then starts uvicorn telecloud.main:app --host 0.0.0.0 --port $PORT.
  6. When live, copy the service URL, e.g. https://telecloud-api.onrender.com — call this the BACKEND URL.
Smoke test: open <BACKEND URL>/health in a browser. You should see {"status":"ok"}. (First hit after idle may take 30–60s — free tier cold start.)

Step 8 — Deploy the frontend to Vercel

8Import the repo and set the API base

  1. Vercel → Add New → Project → import the same GitHub repo.
  2. Root Directory: set to frontend (important — the app isn't at the repo root). Framework auto-detects as Vite; build command npm run build, output dist (already in frontend/vercel.json).
  3. Under Environment Variables, add these three (all public build-time vars):
    NameValueWhy
    VITE_API_BASEthe BACKEND URL from Step 7where the app sends API calls
    VITE_SUPABASE_URLyour SUPABASE_URLenables silent token refresh
    VITE_SUPABASE_ANON_KEYyour anon keyenables silent token refresh
  4. Deploy. Copy the resulting URL, e.g. https://telecloud.vercel.app — call this the FRONTEND URL.
Don't skip the last two vars. With only VITE_API_BASE, the app still works but an expired session (after ~1h) bounces the user to the login screen instead of refreshing silently (frontend/src/config.js → CAN_REFRESH).

Step 9 — Wire the two together (second pass)

9Point the backend at the real frontend URL

Now that you have the FRONTEND URL, go back to Render → telecloud-api → Environment and replace the placeholders:

Env varSet toNotes
APP_BASE_URLthe FRONTEND URLdrives share links + email confirmation redirect. Must be the Vercel URL, not Render. No trailing slash.
CORS_ALLOWED_ORIGINSthe FRONTEND URLcomma-separated if you have several. No trailing slash, no path.

Save — Render redeploys automatically. (Vercel preview URLs like telecloud-abc123.vercel.app are already allowed by a built-in *.vercel.app regex, so you don't need to list every preview.)

Step 10 — Configure Supabase Auth URLs

10So email-confirmation links land on the frontend

Supabase → Authentication → URL Configuration:

  • Site URL = the FRONTEND URL (e.g. https://telecloud.vercel.app).
  • Redirect URLs — add both:
    • https://telecloud.vercel.app
    • https://telecloud.vercel.app/index.html
If these are wrong, the "confirm your email" link either errors or dumps the user on the wrong origin, and login never completes.

Step 11 — Schedule the cleanup jobs (optional)

11QStash → backend cron callbacks

Only needed if you want automated orphan-sweeping / deferred-delete cleanup. Core upload/download/share all work without it.

In Upstash QStash → Schedules → Create, add two schedules that POST to your backend (QStash signs them; the backend verifies with the keys from Step 4):

Destination URLSuggested cron
<BACKEND URL>/jobs/sweep-orphanse.g. hourly 0 * * * *
<BACKEND URL>/jobs/deferred-deletee.g. every 15 min */15 * * * *
On Render's free tier the backend sleeps when idle, so a callback may hit a cold start; QStash retries, so cleanup still runs eventually — just not punctually.

Step 12 — Verify end-to-end

12Acceptance checklist

GET <BACKEND URL>/health → 200 {"status":"ok"}.
Open the FRONTEND URL, sign up → receive Supabase confirmation email → click link → land back on the frontend and reach the dashboard.
Browser DevTools → Network: API calls to the backend return Access-Control-Allow-Origin: <frontend> with no CORS errors in the console.
Upload a file, then download it — the filename is correct (proves Content-Disposition is exposed).
Create a share link, open it in an incognito window — it downloads and shows the correct size (proves Content-Range/Content-Length exposed and the public /s/{token} route works).
Leave the tab idle >1 hour, act again → no forced logout (token refresh works).

Troubleshooting / known hiccups

Everything 500s but /health is fine

You didn't apply the migrations (Step 3), or applied them out of order. Re-run all five; confirm the tables exist in Supabase Table Editor.

First request takes 30–60 seconds

Render free tier spins the service down after ~15 min idle; the next request cold-starts. The frontend already retries with a "waking up" notice — it's a UX delay, not a failure.

App feels slow even when warm

Region mismatch. Put Render, Supabase, and Upstash in the same region. Cross-region round-trips (there are several per request) dominate latency.

CORS errors in the console

Check CORS_ALLOWED_ORIGINS on Render exactly matches the frontend origin: https scheme, host only, no trailing slash, no path. Preview URLs are covered by the built-in *.vercel.app regex; a custom domain must be added explicitly.

Confirmation email never arrives / link is broken

Two causes: (1) Supabase's built-in SMTP is rate-limited to a few emails/hour and often lands in spam — wait and check spam. (2) The Auth URL config (Step 10) is wrong or APP_BASE_URL points at Render instead of Vercel.

Download filename is generic / share shows "0 Bytes"

Expose-headers aren't reaching the browser. This repo already sets expose_headers in the CORS middleware, so if you see this, a proxy or a stale deploy is stripping them — trigger a fresh Render deploy.

Project stopped responding after a week

Supabase free projects auto-pause after ~7 days of inactivity. Open the Supabase dashboard and un-pause. (Render also sleeps, but wakes on the next request.)
